# 基本原則

https://github.com/multica-ai/andrej-karpathy-skills/blob/main/CLAUDE.md を参考に構築

## 1. コーディング前に不明点をすべて消す

実装前に以下を実施する
- 前提を明示する。不確かなら聞く。
- 解釈が複数あるなら提示する。黙って選ばない。
- よりシンプルなアプローチがあるなら伝える。必要なら押し返す。
- 不明な点があったら止まる。何が不明かを言語化して聞く。

## 2. シンプルさを優先する

- 求められた以上の機能は作らない。
- 1箇所しか使わないコードに抽象化は不要。
- 求められていない「柔軟性」や「設定可能性」は持ち込まない。
- あり得ないシナリオへのエラーハンドリングは不要。
- 200行書けて50行で済むなら、書き直す。

## 3. 必要最低限の変更を行う

既存コードを編集する場合、以下に注意する
- 隣接するコード・コメント・フォーマットを改善しない。
- 壊れていないものをリファクタリングしない。
- 自分なら違う書き方をするとしても、既存のスタイルに合わせる。
- 無関係な死んだコードを見つけたら、言及する。削除しない。

自分の変更により既存コードに影響を与えた場合、以下に注意する
- 自分の変更によって不要になったimport・変数・関数は削除する。
- 元からある死んだコードは、求められない限り削除しない。

## 4. ゴール駆動で実行する

タスクをごとに検証可能なゴールを設定し、ゴール達成を心がける。
複数ステップのタスクには、簡潔な計画を示す。
具体例は以下の通り
- 「バリデーションを追加する」→「不正な入力のテストを書いて、通す」
- 「バグを直す」→「再現するテストを書いて、通す」
- 「Xをリファクタリングする」→「前後でテストが通ることを確認する」

# シェルコマンドのツール選択

Rust製の代替のモダンツールがインストール済みです。シェルコマンドでは必ずこちらを使うこと。
モダンツールが存在する場合、レガシーツールを使ってはならない。

- ファイル内容の検索: 必ず `rg`（ripgrep）。`grep` を使わない。
- ファイル名/パスでの検索: 必ず `fd`。`find` を使わない。
- ファイル内の置換: 必ず `sd`。置換目的で `sed` を使わない。

各ツールのネイティブ構文を使うこと。grep/find のフラグをそのまま直訳しないこと。
引数構文が異なるため、機械的な直訳はコマンドを壊す。

構文の対応表（レガシー → モダン）
- `grep -rn "pat" .`                  → `rg "pat"`
- `grep -rn "pat" --include=*.go`     → `rg "pat" -g '*.go'`
- `grep -i "pat" file`                → `rg -i "pat" file`
- `grep -l "pat" .`                   → `rg -l "pat"`
- `find . -name "*.go"`               → `fd -e go`
- `find . -name "foo*"`               → `fd '^foo'`
- `find . -type d -name dist`         → `fd -t d dist`
- `find . -name "*.tmp" -delete`      → `fd -e tmp -X rm`
- `sed -i 's/old/new/g' file`         → `sd 'old' 'new' file`

補足:
- `rg`/`fd` はデフォルトで `.gitignore` を尊重し、隠しファイルをスキップする。対象に含めたい場合: `rg -uu` / `fd -HI`。
- `rg`/`fd`/`sd` が利用できない場合に限り `grep`/`find`/`sed` にフォールバックする。
- 単純な検索では Bash 経由より組み込みの Grep/Glob ツールを優先する。
