# 基本原則

## 1. コーディング前に不明点をすべて消す

実装前に以下を実施する
- 前提を明示する。不確かなら聞く。
- 解釈が複数あるなら提示する。黙って選ばない。
- よりシンプルなアプローチがあるなら伝える。必要なら押し返す。
- 不明な点があったら止まる。何が不明かを言語化して聞く。

## 2. シンプルさを優先する

- 求められた以上の機能は作らない。
- 1箇所しか使わないコードに抽象化は不要。
- 求められていない「柔軟性」や「設定可能性」は持ち込まない。
- あり得ないシナリオへのエラーハンドリングは不要。
- 200行書けて50行で済むなら、書き直す。

## 3. 必要最低限の変更を行う

既存コードを編集する場合、以下に注意する
- 隣接するコード・コメント・フォーマットを改善しない。
- 壊れていないものをリファクタリングしない。
- 自分なら違う書き方をするとしても、既存のスタイルに合わせる。
- 無関係な死んだコードを見つけたら、言及する。削除しない。

自分の変更により既存コードに影響を与えた場合、以下に注意する
- 自分の変更によって不要になったimport・変数・関数は削除する。
- 元からある死んだコードは、求められない限り削除しない。

## 4. ゴール駆動で実行する

タスクをごとに検証可能なゴールを設定し、ゴール達成を心がける。
複数ステップのタスクには、簡潔な計画を示す。
具体例は以下の通り
- 「バリデーションを追加する」→「不正な入力のテストを書いて、通す」
- 「バグを直す」→「再現するテストを書いて、通す」
- 「Xをリファクタリングする」→「前後でテストが通ることを確認する」

## コード方針

### Go

- エラーは必ず明示的に扱う
- 小さく責務の明確な関数を書く
- グローバル状態を避ける
- テーブル駆動テストを基本とする

### TypeScript

- `any` を使わない
- 外部公開APIは明示的な型を定義する
- 複雑な型より読みやすさを優先する
